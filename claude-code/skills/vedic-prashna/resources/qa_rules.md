# Prashna Q&A 追问规则

> **用途**：SKILL.md §Q&A 追问模式的具体执行规则。规定同一张 Prashna 盘上后续小问的处理边界与流程。
> **纪律核心**：追问不换盘、不重起、不越域、不污染主系统。

---

## 1. 追问模式定义

**追问 = 用户已有一张 Prashna 盘(某个 `prashna_<yyyymmdd_HHMM>_<label>/` 目录)，对该盘的判读结果提出的后续问题**。

追问不同于新提问：
- **新提问** = 起新盘（新的提问时刻 + 新的问题 → 新目录）
- **追问** = 复用已有盘、只做判读延伸/澄清

---

## 2. Q&A 只读边界（沙箱化硬约束 #4 兑现）

### 2.1 允许读

- 本次追问所在的 `prashna_*` 目录下所有产物：
  - `structured_prashna.md`（含 Moon 吠陀动向段、可选 Tajika/KP 段）
  - 已生成的判读单 `prashna_judgment_*.md`
  - 已生成的 `qa_*.md` 历史追问记录

### 2.2 禁读

- ❌ 本命 `structured_data.md`（除非本次盘启用了本命交叉——走 cross-natal-policy 的只读边界）
- ❌ 本命 core 报告 `p2a~p5b`
- ❌ 本命 `user_context.md`（承 KN Rao 硬约束 §7 分析阶段禁读盘主背景铁规）
- ❌ **其他 `prashna_*` 目录**（同一用户可能有多张 Prashna 盘；追问只读**本盘所在目录**）
- ❌ 其他 skill 产物（love/career/synastry/rectifier 的输出一律不读）

---

## 3. 追问类型（3 类，处理不同）

### 3.1 澄清型
用户对判读单里某个术语/判据不理解。

**示例**：
- "DK 是什么意思?"
- "为什么 SAV=27 算硬件不足?"
- "什么是 Chandra Kriya?"

**处理**：
- 引用 `resources/moon-policy.md §5 术语翻译规则` 或对应 policy 文件
- 术语当场翻译 + 一句白话解释
- 不重跑任何计算

### 3.2 细化型
用户想要判读单里某个结论的更细数据。

**示例**：
- "那 Dasha 段具体几号开始?"
- "Moon 换 Nakshatra 精确到分吗?"
- "这个 SAV 值排序在 12 宫里第几?"

**处理**：
- 从 `structured_prashna.md` 已有字段读，不重跑
- 若字段不在，明说"该字段本盘未算，需要在 build 时启用某开关"

### 3.3 拓展型（同问题深挖）
用户对同一问题从不同角度追问。

**示例**：
- "那如果我等到 Dasha 换段再问会不会更好?"
- "本命 Dasha 到这段能承受这个结论吗?"（→ 需要本命交叉，见下）
- "Tajika 层怎么看这个?"（→ 需要 Tajika 副层，见下）

**处理**：
- 若涉及**新的判据层**（Tajika / KP / 本命交叉），走 §5 分岔处理

---

## 4. 硬拒绝的追问

以下追问**必须拒绝**，明确告知用户去起新盘：

### 4.1 新问题伪装成追问

**示例**：
- 原盘问"offer 该不该接"，用户追问"那我什么时候结婚?"

**响应模板**：

> "'什么时候结婚' 是一个新问题，不能在'offer 该不该接'的 Prashna 盘上答。
> Prashna 一张盘只答一问。要问结婚，我起一张新盘：现在的时刻+地点+具体结婚问题。要起吗?"

### 4.2 反复重起同一问题

**触发**：用户在 24 小时内就同一问题反复起盘 / 明说"再看看"。

**响应**：走 `question-taxonomy.md §2.5` 拒绝模板："心念不定，盘不可信"。

### 4.3 想切换到 KP（互斥体系）

**触发**：原盘走 Parashari 主判读（默认），用户追问 "用 KP 看看"

**响应模板**：

> "KP 是独立判读体系，不能在同一盘上'追问时切换'——如果要用 KP 判这个问题，需要**重新起一张标注 KP 判读栈的盘**。
> 你是要重起一张 KP 盘（当前时刻起），还是保持 Parashari 判读继续追问?"

---

## 5. 追问时想启用新判据层的分岔处理

### 5.1 追问时想加 Tajika 副层

**允许**：Tajika 是 Parashari 副层，可以在追问时补充。

**流程**：
1. 用户说"用 Tajika 看看这个"
2. 沿用原盘的提问时刻+地点+问题，重跑 `build_prashna_data.py --enable-tajika`，**写到同一目录**（覆盖 `structured_prashna.md`；判读单产物名不同不冲突）
3. 判读单 §六 Tajika 补充参考按 `tajika-optional.md` 规则出

### 5.2 追问时想开本命交叉

**允许**：本命交叉是可选叠加。

**流程**：
1. 用户说"结合我的本命看这个"
2. 沿用原盘的提问时刻+地点+问题，重跑 `build_prashna_data.py --cross-natal <本命路径>`（未来 build 支持该参数时）
3. 判读单 §七 本命交叉裁决按 `cross-natal-policy.md` 规则出

### 5.3 追问时想切换 KP

**拒绝**：见 §4.3。要新盘。

## 6. Q&A 输出格式

**文件命名**：`qa_<主题短标签>.md`，存**本盘所在的 prashna_\* 目录**。

**结构**：

```markdown
# Q&A: <主题>

**关联盘**：<prashna_yyyymmdd_HHMM_label>
**追问时刻**：<YYYY-MM-DD HH:MM>
**追问类型**：[澄清 / 细化 / 拓展]

---

## 问：<用户原问>

## 答：
<白话解读（术语当场翻译）>
<引用 structured_prashna.md 相关字段（不重跑）>
<若涉及新判据层：说明处理方式>

---

**引用字段**：
- structured_prashna.md §<段>：<字段>
- 判读单 §<段>：<结论>
```

**多轮追问**：追加进同一个 `qa_<主题>.md`；不同主题拆不同文件。

---

## 7. 反向禁读（防呆）

**主系统 skill 侧承约束**：即使 core/pro/love/career/synastry/rectifier 未来接入用户会话，它们的 QA 阶段**禁读 `prashna_*` 目录**（沙箱化硬约束 #4）。

**Prashna Q&A 侧的对偶自律**：Prashna Q&A **禁读其他 skill 产物**——包括本命 core 报告、love/career 分析报告等。理由：这些是 Parashari 主体系用别的方法给出的结论，与 Prashna 卜卦语义生态位不同，混用会稀释判读锐度。

**唯一例外**：本命交叉（走 cross-natal-policy 的严格 4 项只读边界）。

- [ ] Q&A 文件是否落在本盘目录、命名 `qa_<主题>.md`？
