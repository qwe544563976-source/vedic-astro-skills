## 分盘视角分离铁律（所有 varga：D9/D10/D4/D5/D7/D12… 通用；引用任何分盘前必读）

分盘断语必须严格分两条线，禁混用、禁不标视角：

- **线A — 分盘内部宫主**：用【分盘自身 Lagna】推该盘宫主表。数据直接读 structured_data「分盘内部宫主表 + 尊贵度（线A）」段（calc 查表，❌ 禁自推）。
  记法：`D10-L10=Saturn，Saturn 落 D10-H*`。D10-L10 由 D10 Lagna 定，**不是**本命 L10。
- **线B — 跨盘参照**：本命行星保留其【D1 宫主职身份】，看它落分盘哪个宫。
  记法：`本命 L2/L11 主 Jupiter，在 D10 落 H9` —— L*/L* **必须显式标注是 D1 视角**。

**四条禁止**：
① 断言"X 是 H* 主"而不说是哪个盘（"Jupiter 是 H9 主" = 错）；
② "L*+L* 落 D*-H*" 而不标 L*/L* 是 D1 身份；
③ 直接拿 D1 宫主职跳到分盘结论；
④ 混用 D1 尊贵度与分盘内部尊贵度而不说明（分盘尊贵读 structured_data 分盘尊贵段）。

**冲突仲裁**：线A、线B 结论不一致 → 分别呈现，禁折衷成"综合来看"。
**输出结构**：逐星深度审计级分盘（如 D9 Step 2.1）走【线A/线B/综合】三段；概述级分盘（D10/D4/D5 概述）的颗粒度**以各版 SKILL 概述指令为准**——Pro 版走三段，开源版精简为线A/线B 各一句。（无论粗细，线A/线B 都必须显式标视角，不因精简而混用。）
**下游消费**：宫位诊断/专题/十大板块等任何下游步骤引用分盘结论（p3b 等）写正文时，必须**沿用其线A/线B标注**——禁止压平成 D1 宫主职、禁止"X是H*主"不标盘；分盘尊贵度用分盘内部值、与 D1 尊贵度分开标。

   是本铁律在 D9 上的一个**实例**；此纪律同等推广到 D10/D4/D5/D7/D12 等**所有更小分盘**。

## 诊断四维度

<!-- ANCHOR: A-HF-321c8936 L1 -->
### 1. 管理者（宫头星/宫主）
<!-- ANCHOR: A-HF-3600cc0f L2 -->
```
查看宫主的：
  P1身份 → 它是什么角色？
  去向(落宫) → 它去了哪个宫位？把这个宫的能量带到了哪里？
  P7尊贵度 → 它健不健康？
  P9 Shadbala → 它有没有力量执行？
```

<!-- ANCHOR: A-HF-5dd9273d L1 -->
### 2. 租客（宫内行星）
<!-- ANCHOR: A-HF-bf8b7681 L2 -->
```
查看宫内星的：
  自带属性 → 吉星进驻=加持，凶星进驻=干扰
  合相情况 → 多颗星同宫=事务繁忙
```

<!-- ANCHOR: A-HF-6d206363 L1 -->
### 3. 相位（谁在看这个宫位）
<!-- ANCHOR: A-HF-cd352345 L2 -->
```
查看哪些行星对此宫位有相位（P10）：
  吉星相位 = 外部加持（保护/增益）
  凶星相位 = 外部压力（冲突/限制/放大）

  相位合成铁律（吉凶相位并存时，沿用"方向/执行分层"模式）：
    分层陈述——吉定结果保底、凶定过程摩擦
    ❌ 禁止择一而论（只报木星保护 或 只报土星限制）
    ❌ 禁止对冲抵消（"一吉一凶互相抵消"）
  多星相位 → 禁止笼统写"受多方面外部影响"，必须按上述铁律
    把每颗相位星归入"结果保底/过程摩擦"两层逐星陈述
```

<!-- ANCHOR: A-HF-623312fe L1 -->
### 4. 硬件带宽（SAV）— 辅助维度

<!-- ANCHOR: A-HF-b320f0dc L2 -->
⚠️ SAV读取铁规（唯一定义，各 skill 引用处逐字同句）：引用任何宫位SAV值，必须从structured_data「宫位映射」表读取；禁止读「原始值（按星座）」表或自行计算sign→house映射；输出标注格式 N宫(Sign)SAV=X。

<!-- ANCHOR: A-HF-133cdd0a L2 -->
```
SAV是辅助确认指标。展示与提及规则全 skill 统一为单点：仅极端值（>32 或 <20）时在数据表列出/在叙事中提及（见 SKILL.md 输出模板与关键原则6），中间值不展开。四档参考：
  SAV > 32 = 溢出（资源充裕）
  SAV 26-32 = 平稳（正常运作）
  SAV 20-25 = 紧张（需要额外努力）
  SAV < 20 = 匮乏（结构性不足）

特殊：凶宫(6/8/12) SAV低 = 反而是防火墙（贼进空屋）
      凶宫(6/8/12) SAV高 = 反转效果（破坏资源充足）
  适用边界：本条仅指该宫灾祸类事件的频率/强度层（6=疾病/官非、8=突变/危机、12=损耗），
    不适用于该宫正面义项（6=竞争/服务力、8=遗产/研究、12=海外/灵性）与掌宫行星的资源评估——
    正面义项与掌宫资源仍按上方四档正读（低=匮乏、高=充裕），
    见p1_p12 P4适用边界与下方大运负面条件5
```

## 宫位评级

<!-- ANCHOR: A-HF-a4b8ad1c L2 -->
```
创始人模式：宫主强+宫内有吉星+吉星相位 → 自主掌控
创始人模式（带阻力）：宫主强+凶星相位 → 仍自主掌控，但过程带阻力（吉凶相位并存时按相位合成铁律分层陈述）
经理人模式：宫主中等+正常相位 → 正常运作
吉祥物模式：宫主弱+有吉星相位 → 有贵人帮但自己弱
飘萍模式：宫主弱+无吉星+凶星相位 → 该领域缺乏根基
（SAV极端值可作为模式判定的辅助确认，但不作为主要判据）
```

### 大运正负面判断规则

大运星的正负面不是"感觉"，是有明确判定条件的：

<!-- ANCHOR: A-HF-84f39d28 L2 -->
⚠️ 节点例外：大运/小运星为 Rahu/Ketu 时，本节六条正负面条件不可计算（节点无掌宫身份/自身尊贵度）——改用 p1_p12「Dasha速查表节点专用正负面条件子表」（P7.4）；速查表节点行"P1身份"列填"—"、"尊贵度"列填定位星状态。

#### 正面条件（满足≥2条→判定为正面）
<!-- ANCHOR: A-HF-9d8e6e64 L2 -->
```
1. P1身份=吉角色（Core-Driver / Yogakaraka / Faithful）
2. P7尊贵度≥友方（Friend/Own/Exalted）
3. P9 Shadbala≥120%
4. 落在吉路（1/4/5/7/9/10宫）
5. 管的宫位SAV≥28
6. Vargottama或Dig Bala加成
```

#### 负面条件（满足任一即标注风险）
<!-- ANCHOR: A-HF-ad7de947 L2 -->
```
1. P1身份=凶角色（Destroyer管8/12 或 Growth-Hacker管6）
2. P2受损（燃烧/落陷/败相）
3. 落在凶路（6/8/12宫）且无VRY补救
4. P9 Shadbala<100%
5. 管的宫位SAV<25（所管宫为凶宫6/8/12时照常计分——计的是该宫正面义项的资源匮乏；
   "凶宫SAV低=防火墙"仅属灾祸事件层，不冲销本条负面分）
6. 与凶星紧密合相（<5°）
```

#### 判定逻辑（按顺序判定，首个命中即停——互斥、全覆盖）
<!-- ANCHOR: A-HF-b30f4990 L2 -->
```
1. 危机期：正面=0 且 负面≥2 → 需谨慎/有风险（最严条件先判）
2. 困难期：正面<2 且 负面≥1 → 该领域面临挑战
3. 混合期：正面≥2 且 负面≥1 → 发展中伴随压力
4. 正面期：正面≥2 且 负面=0 → 该领域顺利/上升
5. 平淡期：以上都不中（即 正面<2 且 负面=0）→ 无显著信号，该领域平稳过渡（兜底）

⚠️ 必须从1往5顺序判，首个命中即定档；禁止乱序读表。
```

### Dasha事件优先级（多事件同期激活时的排序规则）

当用户问"先发生A还是先发生B"时（如"先换工作还是先结婚"），必须用以下规则排序：

<!-- ANCHOR: A-HF-312cb158 L2 -->
```
规则1：Antardasha时序优先
  谁的Antardasha先到，谁的事件先发生。
  例：L10的AD在2026.3-2027.1，L7的AD在2027.1-2027.8
  → 换工作(10宫)先于结婚(7宫)

规则2：同一Antardasha内，大运主管宫优先
  如果两个事件在同一个Antardasha内都被激活：
  大运主(MD lord)管的宫位事务 > 小运主(AD lord)管的宫位事务
  → 大运定基调，与大运主相关的事件先落地

规则3：行星力量排序
  同时激活的两个事件，力量更强的行星（Shadbala更高）先显化。
  强星的事务更容易"冒出来"，弱星的事务需要更多外部催化。

规则4：多阶段事件的阶段分辨
  参照上方"多阶段事件推导"表：
  很多时候A和B并非互斥——可能A处于阶段1（启动），B处于阶段2（落地）。
  必须分辨用户问的是"启动"还是"完成"。
  例："换工作"可能阶段1(机会出现)在2026，阶段2(正式入职)在2027。

输出格式：
  "根据Dasha时间线：
   [事件A]的激活窗口在[AD期间]（[年月]-[年月]）
   [事件B]的激活窗口在[AD期间]（[年月]-[年月]）
   时间上[事件A]先于[事件B]约[X]个月。
   但[事件B]的启动信号(阶段1)可能与[事件A]的落地(阶段2)重叠。"
```

<!-- ANCHOR: A-HF-ecb2e545 L2 -->
⚠️ **禁止**：不能用"感觉"或"一般来说"排序。必须查Dasha时间表给出具体Antardasha期间。
